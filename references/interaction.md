# 交互准则:优先结构化提问

skill 流程里每个"停下来等作者拍板"的环节,都按下面的方式问,别用大段文字罗列让作者自己打字。

## 平台自适应

- **Claude 生态(有 AskUserQuestion 工具)**:用 AskUserQuestion 给结构化选项。把要问的拆成明确选项,作者点选即可。推荐选项排首位并标「(推荐)」。
- **其他平台(Codex 等,无此工具)**:退化为**带编号的明确选项**,让作者回选项号。形如:

  ```
  这一步你想怎么走?
    1) A 方案(推荐)
    2) B 方案
    3) 其他(直接说)
  ```

  不要开放式长问,也不要大段输出占屏。

## 哪些环节要给选项

blog 的关键停顿点都适用:

- **定题**:边界收/扩、visibility(public / private)——给选项;主题方向这类开放表达,允许作者补充文字。
- **采集**:素材源(当前对话 / git / llmdoc / 口述)用多选勾选。
- **大纲**:确认 / 调整 / 直接出——给选项。
- **去味**:跑一遍 / 跳过——给选项。

## 两条边界

- **开放表达仍允许**。给选项不等于堵死自由输入——主题边界、补充素材这类,选项之外留一个"其他/补充"口子。
- **别为问而问**。能从上下文合理推断的默认值,直接用并说明,不必每步都弹问。只在作者的回答会改变下一步做法时才停下来问。
