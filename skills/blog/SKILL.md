# blog:写一篇开发技术随笔

从开发素材(对话、git、llmdoc、口述)写出一篇聚焦单一主题的中文技术随笔。一篇只讲一件事。

默认**分步**走,在大纲处停一下等作者确认。作者说"别问了直接出""一气呵成"之类,就跳过确认、一路写到归档。

每个等作者拍板的环节,先读交互准则,按它的方式给选项(Claude 下用 AskUserQuestion,其他平台给编号选项),别用大段文字让作者打字。读插件根目录下的 `references/interaction.md`;从本 skill 目录解析时路径是 `../../references/interaction.md`。

## 流程

1. **定题**。和作者确认这篇讲什么、边界在哪,避免宽泛。题太大就切小。边界收/扩、visibility 给选项;主题方向留补充口子。
2. **采集**。按 `references/collect.md` 列出可用素材源(当前对话 / git / llmdoc / 作者口述),用多选让作者勾选,可补充。只取与主题相关的。
3. **大纲**。给一份要点骨架,标清方案对比、踩坑、心得放在哪。**默认在此停下**,以选项形式让作者确认 / 调整 / 直接出;若作者要"直接出"则跳过此停顿。
4. **成稿**。按 `references/narrative.md` 的写时基调和叙事方式写,不套固定骨架。基调在动笔时就立,治结构性 AI 味。
5. **去味**。以选项形式问作者跑一遍还是跳过。要跑就读去 AI 味清单(本 skill 目录下的 `../humanize/references/ai-tells.md`),对草稿做一遍审查改写:逐条对照修改。这一步治措辞/语气/局部排版的表皮残留;结构问题应在第 4 步就避免。

6. **归档**。按 `references/archive.md` 写入 `.blog/YYYY-MM-DD/<主题-slug>.md`(不静默覆盖),报告落盘路径。

## 注意

- 事实(数字、文件名、报错)以 git 和实际输出为准,不编。拿不准的问作者。
- 这是技术随笔,不是产品文案。少套话、少营销腔、少无谓加粗。
