# 叙事模板 + 写时基调

## 写时基调(动笔前先立,治结构性 AI 味)

带着这几条写,初稿就不会是模板骨架。这是**正向定调**,不是禁令清单——放开写,只是记着这个调子:

1. 像一个具体的人在写技术随笔,记给同行(或未来的自己)看,不是写产品文案。
2. 老实写过程:不确定的、走过的弯路、还没搞懂的,都可以留在文里。比"一切尽在掌握"的假象可信。
3. 可以诙谐,可以有情绪(被坑到的吐槽、想通时的痛快),但别硬抖机灵。
4. 让句子长短交替、段落节奏不一,别每段一个模子。
5. 有判断就给判断——"我觉得这样更好,因为……",别只罗列客观信息。

## 结构(参考,不是模板)

技术随笔不套固定骨架。常见的几种顺序,挑合适的,别硬凑:

- **问题驱动**:遇到什么问题 → 试了哪些路 → 为什么选这条 → 结果怎样 → 留了什么坑。
- **决策复盘**:摆出几个方案 → 各自利弊 → 当时怎么权衡 → 现在回看对不对。
- **踩坑记录**:现象 → 排查过程 → 真正的原因 → 怎么绕过/根治 → 教训。

方案对比、踩坑、心得这些料,放进上面顺序里自然的位置,不必单设"方案对比""踩坑总结""心得体会"三个并列大标题——那本身就是一种模板感。

## 标题怎么起

标题要精准,也要自然有劲,别把主题词原样照搬当标题——那很死板。

- 照搬:主题"给 skill 加交互准则" → 标题就写"给 skill 加交互准则"。死。
- 有劲:同一篇 → "遇事别堆字,给选项"。点出文章真正的劲在哪。

起法:想清楚这篇最想说的那句话/那个转折是什么,用口语把它点出来。可以有画面感、有判断、有点情绪,但别硬凑金句、别夸大。topic 字段留原主题词管检索,title 管让人想点进来。

## 详略(跟着 visibility 走)

- `public`:面向不认识作者的读者,必要背景要交代清楚,起承转合完整。
- `private`:给未来的自己/团队,背景能省则省,语气更随意,可以只记关键决策和坑。
